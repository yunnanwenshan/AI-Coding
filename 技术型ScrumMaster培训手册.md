## 0. 使用说明

**本文档是培训手册的大纲版**，讲"11 个节点分别是什么、做什么、产出什么"。具体怎么落地（Checklist、案例、话术、错误示范）放在附录 A 的模板 + 附录 C 的展开资料索引中。

**使用方式**
- 上任/接手新团队：先通读第 1-2 节，定位"我现在卡在哪个节点"
- 日常 Sprint：每周打开第 2 节，对照当前所在的节点看产出物
- Sprint 复盘前：对照第 2 节每节点的"常见错误"，检查本 Sprint 命中了几条
- 培训新人：让新人先读本文档，再按附录 C 索引去读具体节点展开

**与配套文档的关系**
- 《技术型 Scrum Master / 小组长 能力模型》——评价层，讲"我在哪一级 / 该往哪走"
- 本文档——执行层大纲，讲"11 个节点做什么 / 产出什么"
- 附录 A 模板——工具层，可直接复制使用

## 1. 角色定位与边界

**一句话定位**：懂业务、懂技术、控节奏、清障碍、保质量、带团队。

**三个常见误解**
- ❌ "SM 就是会议主持人" → 主持人是表层动作，核心是清障碍 + 控风险
- ❌ "技术型 SM 等于半个架构师" → 能看懂方案 + 识别风险，但技术决策仍归技术负责人
- ❌ "SM 替产品做决策" → 不替代需求价值与优先级，但能质疑需求清晰度

**职责边界速记**

| 事项 | SM | PO | 技术负责人 |
|---|---|---|---|
| 需求价值/优先级 | 不决策 | 主决策 | 评估 |
| 需求清晰度/验收 | 推动补齐 | 主责 | 评估 |
| 技术方案设计 | 参与评审 | 不参与 | 主决策 |
| 技术风险识别 | 主识别 + 闭环 | 不参与 | 复核 |
| Sprint 目标承诺 | 主组织 | 参与 | 参与 |
| 进度跟踪/预警 | 主负责 | 关注业务 | 关注技术 |
| 跨团队清障 | 主负责 | 协同 | 协同 |
| 团队能力提升 | 主负责 | 参与 | 协同 |

## 2. 完整生命周期：11 个关键节点

```
[1 需求进入] → [2 技术评审] → [3 Sprint 计划] → [4 执行+日站会]
                                                     ↓
[11 团队提升] ← [10 复盘改进] ← [9 Sprint 评审] ← [8 上线观察] ← [7 发布检查] ← [6 联调验收] ← [5 质量把关]
```

### 节点速查表

| # | 节点 | 核心动作 | 关键产出物 | 关注指标 | 常见错误 |
|---|---|---|---|---|---|
| 1 | 需求进入与预评审 | 挡不清晰的需求在 Sprint 外 | 需求准入 Checklist（A1） | 需求准备度 ≥ 80%；Sprint 中需求变更 ≤ 20% | 预审变成"再聊聊" |
| 2 | 技术方案与风险评审 | 编码前识别 8 项技术风险 | 技术方案评审 Checklist（A2） | 方案评审覆盖率 ≥ 90%；重大风险提前识别 ≥ 2/Sprint | 评审会变成念 PPT |
| 3 | Sprint 目标与计划制定 | 围绕"目标"承诺，不围绕"任务" | Sprint 目标卡（A3） | 目标清晰度 ≥ 90%；估算偏差 0.8-1.2 | 没有目标直接排任务 |
| 4 | Sprint 执行与每日节奏 | 看板 + 站会 + 中期检查 3 件套 | Sprint 看板（A4）+ 阻塞跟踪表（A7） | 任务按期率 ≥ 80%；阻塞处理 ≤ 24h；WIP ≤ 人数×1.5 | 站会变汇报会 |
| 5 | 技术质量过程把关 | 把问题消灭在编码阶段 | 代码评审 + 自测 + 扫描 | CR 覆盖 100%；缺陷前移率 ≥ 40%；提测一次通过 ≥ 70% | 自测不写用例 |
| 6 | 联调测试与验收推进 | 缺陷在测试阶段闭环 | 提测评审 + 缺陷 4 级 + 验收清单 | P0 ≤ 4h / P1 ≤ 24h / P2 ≤ 48h；验收一次通过 ≥ 70% | 验收走形式 |
| 7 | 上线准备与发布检查 | 可灰度 / 可回滚 / 可监控 | 上线 Checklist（A5） | Checklist 完成 100%；发布失败 ≤ 5%；回滚 ≤ 1/季度 | 没有回滚方案 |
| 8 | 上线观察与问题响应 | 24h 观察 + 故障分级响应 | 问题清单 + 复盘记录 | P0 ≤ 30min / P1 ≤ 4h MTTR；重复问题下降 | 上线后"放羊" |
| 9 | Sprint 评审与目标验收 | 验证"目标达成"而非"任务完成" | 目标达成表 + 业务签字 | 目标达成率 ≥ 80%；业务验收签字 100% | 任务完成 = 目标达成 |
| 10 | 回顾复盘与改进闭环 | 3-5 个 SMART 改进项 | 改进项跟踪表 | 改进项闭环率 ≥ 80%；重复问题下降 | 复盘变吐槽大会 |
| 11 | 知识沉淀与团队提升 | 月分享 + 师傅带教 + 知识地图 | 文档库 + 知识地图 | 文档 ≥ 2/月；分享 ≥ 1/月；新人上手 ≤ 4 周 | 分享会变 PPT 朗读 |

**11 节点共性要求**
- 每个节点都遵循"方法 + 案例 + 落地动作 + 指标 + 常见错误"五件套（详见附录 C 展开资料）
- 每节点 ERP/财务域特殊关注：节点 1（金蝶对接）、2（批量结账）、5（红字冲销）、7（年度结转）、8（月末结账窗口）
- 每节点失败信号：上一节点的产出物缺失 / 上一节点的指标超阈值

## 3. 节点串联：从单 Sprint 到季度规划

```
单 Sprint 闭环（2 周）：
节点 1-2（预审）→ 3-4（计划/执行）→ 5-6（质量/联调）→ 7-8（发布/观察）→ 9-10（评审/复盘）
                                                                                        ↓
                                                                            节点 11（沉淀）形成长期资产
                                                                                        ↓
季度规划（13 周）：
每月 1 次"季度中期复盘"（节点 10 延伸）→ 季度末"季度回顾 + 下季规划" → 跨季度技术债治理
```

**季度层面 3 件事**
1. 月度数据复盘：Sprint 目标达成率、缺陷密度、MTTR、改进项闭环率的月度趋势
2. 季度技术债治理：从复盘中提取 Top 3 技术债，安排专门 Sprint 治理
3. 团队能力盘点：对照分级标准，识别初中高级人才梯队

## 4. 分级关注重点

| 节点 | 初级 SM 重点 | 中级 SM 重点 | 高级 SM 重点 |
|---|---|---|---|
| 1 需求进入 | 按 Checklist 执行 | 推动预评审机制 | 建立需求准入标准 |
| 2 方案评审 | 参与评审 | 主导评审 | 建立评审机制 + 评审官梯队 |
| 3 Sprint 计划 | 按规范开计划会 | 设定 Sprint 目标 | 建立目标管理体系 |
| 4 执行节奏 | 维护看板 + 站会 | 识别跨团队风险 | 建立跨团队交付机制 |
| 5 质量把关 | 执行代码评审 | 推动工程实践 | 建立质量体系 |
| 6 联调验收 | 跟踪缺陷 | 推动缺陷复盘 | 建立缺陷分析机制 |
| 7 发布检查 | 执行 Checklist | 推动灰度机制 | 建立发布治理 |
| 8 上线观察 | 跟踪告警 | 推动问题复盘 | 建立稳定性体系 |
| 9 Sprint 评审 | 组织评审会 | 设定目标验收标准 | 建立目标管理方法论 |
| 10 复盘改进 | 组织复盘会 | 推动改进闭环 | 建立持续改进机制 |
| 11 团队提升 | 组织分享会 | 培养骨干 | 建立人才梯队 + 方法论输出 |

**关键差异**
- 初级：执行 + 单一团队，**不**要求建立机制
- 中级：复杂团队 + 跨团队，**开始建立机制**
- 高级：跨业务线 + 组织级，**输出方法论 + 赋能多个团队**

## 5. 落地路径：本季度可以做哪些事

| 阶段 | 时间 | 重点动作 |
|---|---|---|
| 第 1 个月（立即做） | 第 1-4 周 | 需求准入 Checklist（节点 1）+ 技术方案评审 Checklist（节点 2）+ 看板 + 阻塞表（节点 4）+ 上线 Checklist（节点 7） |
| 第 2 个月（机制建设） | 第 5-8 周 | 代码评审 100% 覆盖（节点 5）+ 复盘会 3-5 项改进上限（节点 10）+ 每月 1 次技术分享（节点 11） |
| 第 3 个月（体系化） | 第 9-12 周 | 月度数据复盘机制 + 新人带教制度 + 团队知识地图 |
| 第 4 个月（方法论） | 第 13-16 周 | 输出"Sprint 节奏手册 v1.0"+ 启动技术债治理专项 + 培养 SM 梯队 |

## 6. 常见反对意见与回应

| 反对意见 | 回应 |
|---|---|
| "Sprint 才 2 周，搞 11 节点太重" | 节点是**视角**不是会议，多数节点融合在已有活动中，是 11 个 Check 时刻 |
| "团队 5 人，文档沉淀没必要" | 5 人团队文档沉淀收益更高（人员流动冲击小），用最简模板 1 小时可完成一篇 |
| "上线 Checklist 让发布变慢" | 短期看变慢，长期看故障率下降 + 救火时间减少，一次 P0 故障损失 = 10+ Checklist 工时 |
| "技术型 SM 抢了技术负责人的活" | 边界明确：技术负责人**决策**，SM **识别风险 + 推动评审 + 跟踪闭环**，是"风险雷达"不是"决策者" |
| "复盘会没用，问题下次还出" | 90% 是改进项无闭环，不是复盘会没用。强制 3-5 项 + 责任人 + 完成时间 + 下 Sprint 评审，闭环率可达 80%+ |

## 7. 总结：技术型 SM 的 6 个核心动作

无论节点再多，技术型 SM 的 6 个核心动作是：

1. **挡需求**（节点 1）：不清晰的需求不进 Sprint
2. **审方案**（节点 2）：复杂需求必过技术评审
3. **盯节奏**（节点 4）：看板 + 站会 + 中期检查
4. **卡质量**（节点 5-6）：代码评审 + 提测门槛 + 缺陷分级
5. **管发布**（节点 7-8）：Checklist + 灰度 + 24h 观察
6. **促改进**（节点 9-11）：评审 + 复盘 + 沉淀

把这 6 件事做扎实，团队交付就从"按任务推进"升级为"按目标稳定交付"。
