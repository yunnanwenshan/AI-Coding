# 技术型 Scrum Master / 小组长 能力模型

版本：v1.0（定稿）
适用对象：技术型 Scrum Master、技术型小组长、HR/HRBP（定级与晋升评审参考）
来源：基于一线敏捷实践沉淀，结合工程效能与组织敏捷成熟度要求

---

## 0. 背景：这份文档解决什么问题

公司在多业务线并行迭代（ERP / 电商中台 / 财务核算）的背景下，一线技术型 Scrum Master / 小组长普遍存在三类偏差：

1. 只做流程组织，不会识别技术风险，团队"按任务推进"但质量不稳定。
2. 技术能力有余但节奏管理、跨角色清障能力不足，Sprint 目标难以稳定达成。
3. 同一岗位做三五年仍无法清晰对标"我现在是初级/中级/高级"，晋升和定级缺乏客观依据。

本模型用一套"能力 + 职责 + 指标 + 边界 + 分级"五位一体的标准，回答四个问题：
- 这个岗位要做什么、不做什么？
- 做得好不好用什么指标衡量？
- 不同级别的人在能力上具体差在哪里？
- 候选人/现任怎么用这份模型自我对标、准备晋升举证？

## 1. 岗位定位

技术型 Scrum Master / 小组长是一线敏捷团队的 **交付节奏负责人 + 技术质量协同负责人 + 持续改进推动者**。该岗位不是传统意义上只负责流程组织的 Scrum Master，而是必须具备高级工程师能力，能够在组织 Sprint 交付节奏的同时，识别技术风险、系统稳定性风险、工程质量问题和跨角色协作障碍，推动团队实现 **稳定、高质量、可持续交付**。

### 1.1 一句话定位

**懂业务、懂技术、控节奏、清障碍、保质量、带团队。**

### 1.2 与传统 Scrum Master 的本质差异

| 维度 | 传统 SM | 技术型 SM / 小组长 |
|---|---|---|
| 核心交付物 | 流程运转 + 团队协同 | 流程运转 + 技术风险闭环 + 持续改进 |
| 技术介入深度 | 不参与技术决策 | 能看懂方案、识别风险、推动责任人闭环 |
| 质量责任 | 推动测试验收 | 推动代码评审、技术评审、工程实践、稳定性体系 |
| 价值定位 | 会议主持人 / 团队教练 | 团队交付的"稳定器"+"质量守门人"+"改进发动机" |

### 1.3 三个常见误解

- ❌ "Scrum Master 就是会议主持人" → 主持人是表层动作，核心是清障碍 + 控风险。
- ❌ "技术型 SM 等于半个架构师" → 能看懂方案 + 识别风险，但 **技术决策仍归技术负责人**。
- ❌ "Scrum Master 替产品经理做决策" → 不替代 **需求价值与优先级** 决策，但能质疑需求清晰度和验收标准。

## 2. 核心职责

技术型 SM / 小组长的核心职责收敛为 **三大板块**，覆盖交付节奏、技术质量、协同清障 + 团队提升：

### 2.1 敏捷交付与节奏管理

围绕 Sprint 目标组织需求拆解、任务计划、工作量评估、进度跟踪、风险预警、评审验收和回顾复盘，保障团队形成稳定、透明、可控的迭代交付节奏。

**关键动作**：
- 推动产品、开发、测试在迭代开始前明确需求范围、验收标准和交付边界。
- 控制需求变更和临时插入对 Sprint 的影响。
- 及时识别延期风险、资源冲突和交付阻塞，推动问题闭环。

### 2.2 技术把关与质量保障

具备高级工程师能力，能够参与技术方案评审、核心模块设计、接口设计、数据模型设计和复杂业务逻辑分析，识别事务处理、并发控制、缓存、消息机制、分布式调用、权限、安全、性能、数据一致性等关键技术风险。

**关键动作**：
- 协同技术负责人推动代码评审、测试验收、上线检查。
- 主导/参与缺陷复盘、线上问题复盘、技术债治理。
- 推动工程实践落地（CI/CD、代码扫描、自动化测试、接口规范、日志规范）。

### 2.3 协同清障与团队提升

协调产品、研发、测试、运维、业务等角色，推动需求不清、依赖阻塞、环境问题、资源冲突、质量问题和跨部门问题及时解决，减少等待、返工和无效沟通。

**关键动作**：
- 通过技术分享、问题案例沉淀、新人带教、项目复盘和最佳实践复用。
- 提升团队整体交付能力、工程规范意识和工程能力。
- 推动团队从被动执行逐步走向主动协同和自我改进。

## 3. 核心能力

五大能力维度，对应第 2 节的三大职责：

| 能力维度 | 对应职责 | 一句话定义 |
|---|---|---|
| 1. 高级工程师能力 | 技术把关 | 能看懂方案、识别风险、指导团队解决复杂问题 |
| 2. 敏捷交付组织能力 | 节奏管理 | 能组织会议、推动目标达成、控制节奏透明可控 |
| 3. 技术质量与工程实践能力 | 质量保障 | 能推动工程实践落地、治理技术债、保障系统稳定性 |
| 4. 协同清障与风险预警能力 | 协同清障 | 能识别跨角色协作卡点、推动升级、提前预警 |
| 5. 团队培养与持续改进能力 | 团队提升 | 能通过复盘、分享、带教、标准化沉淀提升团队能力 |

### 3.1 高级工程师能力

具备复杂业务建模、接口设计、数据模型设计、系统拆分和核心模块开发指导能力，能够识别性能、稳定性、安全性和数据一致性风险，能看懂技术方案、定位关键问题并指导团队解决。

**典型行为**：
- 能独立完成中等/复杂业务模块的设计、开发和上线。
- 能识别事务、并发、缓存、消息、分布式调用中的关键风险。
- 能在跨团队、跨系统场景下主导关键技术攻关。

### 3.2 敏捷交付组织能力

能够组织计划会、站会、评审会和回顾会，推动团队明确 Sprint 目标、任务拆解、交付节奏和验收边界，保障迭代过程透明可控。

**典型行为**：
- 能按规范组织四会，控制会议时长（计划会 90-120 分钟、站会 ≤ 15 分钟、评审会 60-90 分钟、回顾会 60-90 分钟）。
- 能维护 Sprint 看板，跟踪任务状态、进度和风险。
- 能识别进度偏差、需求变更、临时插入的负面影响并推动调整。

### 3.3 技术质量与工程实践能力

能够推动代码评审、接口规范、测试验收、自动化测试、代码扫描、CI/CD、上线检查、缺陷复盘和技术债治理等工程实践落地。

**典型行为**：
- 能建立并维护工程规范、接口规范、日志规范。
- 能推动自动化测试覆盖核心路径和异常路径。
- 能组织缺陷复盘和线上问题复盘，推动改进项落地。

### 3.4 协同清障与风险预警能力

能够识别并推动解决产品、研发、测试、运维、业务之间的协作问题和交付风险，提前暴露问题并推动闭环。

**典型行为**：
- 能识别需求不清、依赖阻塞、环境问题、资源冲突、质量问题等常见障碍。
- 能推动跨部门问题升级和闭环，提前预警交付风险。
- 能从流程、组织、工具层面系统性识别和解决协作障碍。

### 3.5 团队培养与持续改进能力

能够通过复盘、分享、带教和标准化沉淀，提升团队技术能力、质量意识和自组织能力。

**典型行为**：
- 能组织复盘会并跟进改进项落地。
- 能进行新人带教和基础技术分享。
- 能沉淀技术方案、问题案例和最佳实践。
- 能建立团队持续改进和知识管理体系。

## 4. 关键考核指标

### 4.1 核心指标（必看三项）

| 序号 | 指标 | 衡量内容 | 关注维度 |
|---|---|---|---|
| 1 | **Sprint 目标达成率** | 团队是否围绕 Sprint 目标稳定交付 | 目标完成、延期预警、复盘改进 |
| 2 | **交付质量与系统稳定性** | 上线后质量表现 | 缺陷数、严重故障、回滚、重复问题、质量改进 |
| 3 | **技术风险识别与闭环效果** | 技术把关是否有效 | 方案评审、核心模块风险、性能/稳定性预警、技术债治理、复杂问题解决 |

### 4.2 辅助观察指标

- **工程实践落地情况**：CI/CD、代码扫描、自动化测试、代码规范等机制的建立和执行。
- **团队能力提升情况**：新人上手周期、骨干培养、分享次数、文档沉淀。

### 4.3 指标采集建议

| 指标类别 | 采集方式 | 采集频率 |
|---|---|---|
| Sprint 目标达成率 | Sprint 评审会 + 目标达成表 | 每次 Sprint |
| 缺陷数 / 严重故障 / 回滚 | 缺陷管理工具 + 上线记录 | 每周 / 每次发布 |
| 风险识别数 | 方案评审记录 + 复盘记录 | 每次评审 / 每次复盘 |
| 工程实践落地 | 工具平台 + Check 清单 | 每月 |
| 团队能力提升 | 知识库 + 分享记录 | 每季度 |

## 5. 岗位边界

技术型 SM / 小组长 **不替代** 产品经理负责需求价值和优先级决策；但 **必须能** 看懂技术方案、判断技术风险、推动责任人闭环，并在必要时指导团队完成关键技术攻关。

### 5.1 职责边界对照表

| 事项 | SM / 小组长 | 产品经理 | 技术负责人 |
|---|---|---|---|
| 需求价值和优先级 | 不决策 | 主决策 | 参与评估 |
| 需求清晰度、验收标准 | 推动补齐 | 主责 | 参与评估 |
| 技术方案设计 | 参与评审 | 不参与 | 主决策 |
| 技术风险识别 | 主识别 + 推动闭环 | 不参与 | 复核 + 决策 |
| Sprint 目标承诺 | 主组织 | 参与 | 参与 |
| 进度跟踪和风险预警 | 主负责 | 关注业务 | 关注技术 |
| 跨团队清障 | 主负责 | 协同 | 协同 |
| 团队能力提升 | 主负责 | 参与 | 协同 |

### 5.2 三类越权风险

- ❌ **越权做产品决策**：替产品经理决定做不做、先后顺序。
- ❌ **越权做技术决策**：替技术负责人定技术方案、选型。
- ✅ **正确的姿态**：看得懂、提得出、推动责任人闭环、必要时指导攻关，但 **不替决策**。

## 6. 分级判断标准

本岗位分为 **初级、中级、高级** 三个级别。三级之间在"管多大事 / 解决多深的问题 / 影响多大范围"上有明显差异，下方按能力维度给出 **必须满足 + 可选要求**，便于 HR 定级和晋升评估时对照使用。

### 6.1 整体差异说明

- **初级**：聚焦单一团队交付，能稳定执行敏捷机制和工程规范，能独立解决模块级技术问题。
- **中级**：聚焦复杂业务团队或多个并行团队交付，能独立处理技术方案风险和跨团队协作问题，能推动团队从"能交付"走向"稳定交付"。
- **高级**：聚焦跨团队、跨业务线交付体系与工程效能建设，能主导架构级技术风险解决，能输出方法论并赋能多个团队。

### 6.2 初级 Scrum Master / 小组长

**必须满足（5 项 × 5 条 = 25 条全满足）**：

| 能力 | 必须满足项 |
|---|---|
| 1. 高级工程师能力 | - 能独立完成中等复杂度模块的设计、开发和上线<br>- 能识别常见的性能、数据一致性、稳定性风险<br>- 能指导初中级工程师解决常见技术问题 |
| 2. 敏捷交付组织能力 | - 能按规范组织计划会、站会、评审会和回顾会<br>- 能维护 Sprint 看板，跟踪任务状态和进度<br>- 能发现明显的延期和阻塞问题并推动闭环 |
| 3. 技术质量与工程实践能力 | - 能按规范执行代码评审、测试验收和上线检查<br>- 能维护基本的工程规范、接口规范和日志规范 |
| 4. 协同清障与风险预警能力 | - 能识别需求不清、依赖阻塞、环境问题等常见障碍<br>- 能推动问题在团队内闭环 |
| 5. 团队培养与持续改进能力 | - 能组织复盘会并跟进改进项落地<br>- 能进行新人带教和基础技术分享 |

**可选要求（满足任意 1-2 项即可）**：
- 推动过 1-2 项工程实践落地（如 CI/CD、代码扫描、自动化测试）。
- 主导过 1 个中等复杂度需求的全流程交付。
- 独立推动过跨职能协作中的阻塞问题闭环。
- 沉淀过技术方案、问题案例或团队规范文档。

### 6.3 中级 Scrum Master / 小组长

**必须满足（5 项 × 6 条 = 30 条全满足）**：

| 能力 | 必须满足项 |
|---|---|
| 1. 高级工程师能力 | - 能独立负责复杂业务模块的设计、开发和上线<br>- 能主导技术方案评审，发现并解决架构级技术风险<br>- 能指导团队处理高并发、分布式、复杂业务场景下的关键问题 |
| 2. 敏捷交付组织能力 | - 能组织团队明确 Sprint 目标、需求范围和验收标准<br>- 能识别迭代过程中的进度风险、资源冲突和依赖阻塞<br>- 能推动迭代按节奏完成，控制临时插入对 Sprint 的影响 |
| 3. 技术质量与工程实践能力 | - 能推动团队建立代码评审、测试覆盖、CI/CD、代码扫描等工程实践机制<br>- 能组织缺陷复盘和线上问题复盘，推动改进项落地 |
| 4. 协同清障与风险预警能力 | - 能识别跨产品、研发、测试、业务之间的协作卡点<br>- 能推动跨部门问题升级和闭环，提前预警交付风险 |
| 5. 团队培养与持续改进能力 | - 能沉淀技术方案、问题案例和最佳实践<br>- 能培养初中级工程师，推动团队质量意识和工程能力提升 |

**可选要求（满足任意 2-3 项即可）**：
- 主导过 2-3 个跨团队项目交付。
- 推动过技术债治理或工程质量体系优化。
- 在多团队协作中建立过问题升级或风险预警机制。
- 组织过 3 次以上跨团队复盘或质量改进项目。

### 6.4 高级 Scrum Master / 小组长

**必须满足（5 项 × 6 条 = 30 条全满足）**：

| 能力 | 必须满足项 |
|---|---|
| 1. 高级工程师能力 | - 能主导核心业务建模、关键模块拆分和系统稳定性设计<br>- 能识别架构级和系统级风险，推动技术方案优化和技术债治理<br>- 能在跨团队、跨系统场景下主导关键技术攻关 |
| 2. 敏捷交付组织能力 | - 能为多团队或复杂项目建立稳定的交付节奏和协作机制<br>- 能提前识别跨团队依赖和系统级交付风险<br>- 能持续优化团队迭代效率，提升整体交付可预测性 |
| 3. 技术质量与工程实践能力 | - 能主导工程质量体系和技术债治理机制建设<br>- 能推动团队从"完成交付"走向"高质量稳定交付"<br>- 能将质量改进经验沉淀为团队标准和最佳实践 |
| 4. 协同清障与风险预警能力 | - 能建立跨团队协作和问题升级机制<br>- 能从流程、组织、工具层面系统性识别和解决协作障碍<br>- 能提升整体交付过程的透明度和可预测性 |
| 5. 团队培养与持续改进能力 | - 能推动团队从被动执行走向自组织协同<br>- 能建立团队持续改进和知识管理体系<br>- 能赋能多个团队，提升整体组织敏捷成熟度 |

**可选要求（满足任意 2-3 项即可）**：
- 主导过多个团队或部门的研发效能改进项目。
- 推动过组织级工程实践或敏捷机制落地。
- 在跨部门或跨业务线项目中担任关键技术交付负责人。
- 输出过方法论、流程规范或团队建设最佳实践。
- 培养过多名初中级 Scrum Master 或技术骨干。

## 7. 三级差异对照（供 HR 面试/评审参考）

| 维度 | 初级 | 中级 | 高级 |
|---|---|---|---|
| **管多大事** | 单一团队 / 单业务线交付 | 复杂业务团队 / 多个并行团队交付 | 跨业务线 / 跨部门交付体系建设 |
| **解决多深的问题** | 模块级技术问题、团队内协作问题 | 架构级技术问题、跨团队协作问题 | 系统级 / 架构级技术问题、组织级协作机制问题 |
| **影响多大范围** | 影响本团队交付质量和节奏 | 影响多个团队的交付可预测性 | 影响组织级研发效能、敏捷成熟度和工程文化 |

### 7.1 三级能力的递进逻辑

```
初级：执行 + 单一团队
   ↓ [推动"能交付 → 稳定交付"]
中级：机制 + 复杂团队
   ↓ [推动"稳定交付 → 体系化建设"]
高级：方法论 + 跨业务线 / 组织
```

### 7.2 关键跃迁信号（晋升评估时关注）

**从初级 → 中级，看这 3 个信号**：
- 能独立处理跨团队协作问题，不再只靠"上级出面"。
- 能推动团队从"任务完成"走向"目标达成"。
- 主动建立工程实践机制（CI/CD、代码扫描、自动化测试），不再只执行。

**从中级 → 高级，看这 3 个信号**：
- 能主导跨业务线、跨部门的研发效能改进项目。
- 能输出可被多个团队复用的方法论、流程规范。
- 培养出至少 1 名可独立带团队的初中级 SM / 技术骨干。

### 8.2 晋升举证模板（晋升中级/高级时提交）

```
【晋升举证】候选人：____ 申报级别：____ 评估周期：____

1. 高级工程师能力举证
   - 主导/参与过的复杂模块：____
   - 主导过的技术评审：____
   - 解决过的关键技术问题：____

2. 敏捷交付组织能力举证
   - 主导过的 Sprint 数：____
   - Sprint 目标达成率：____
   - 推动过的跨团队协作案例：____

3. 技术质量与工程实践能力举证
   - 推动过的工程实践：____
   - 主导过的复盘：____
   - 治理过的技术债：____

4. 协同清障与风险预警能力举证
   - 推动跨部门升级的案例：____
   - 提前预警并避免的风险案例：____

5. 团队培养与持续改进能力举证
   - 沉淀的文档清单：____
   - 主导的分享场次：____
   - 培养的骨干：____

6. 核心指标表现
   - Sprint 目标达成率：____
   - 缺陷/故障/回滚数：____
   - 技术风险识别/闭环率：____
```

### 8.3 HR 评估流程建议

1. **自评**：候选人提交诊断清单打分 + 晋升举证材料。
2. **同行评估**：2-3 名同级别或上下级同事匿名打分。
3. **上级评估**：直属上级打分 + 写出 3 条具体案例证据。
4. **答辩**：HR + 业务负责人组织 30-60 分钟答辩，重点考察"具体做了什么、怎么做的、效果如何"。
5. **定级**：HR 综合自评/同行/上级/答辩结果，对照本模型 6.2/6.3/6.4 必须满足项和可选要求，给出最终定级建议。
